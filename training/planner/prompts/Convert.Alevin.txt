### Core Purpose
Convert.Alevin is a **data format conversion module** that bridges two major computational ecosystems in single-cell RNA-seq (scRNA-seq) analysis:

1. **Salmon/Alevin** – a fast, alignment-free, UMI-aware quantification tool that produces binary output files
2. **scVelo/scanpy/AnnData** – the Python-based single-cell analysis ecosystem requiring HDF5-based `.h5ad` or `.loom` files

Specifically, the module is designed to convert **Alevin Velocity quantification output** — which simultaneously quantifies both spliced (exonic) and unspliced (intronic) RNA — into an `AnnData` object (`H5AD` or `Loom` format) that `scVelo` can directly consume for **RNA velocity analysis**.

## 2. Biological Background & Rationale

### RNA Velocity — The Scientific Motivation

**RNA velocity** (La Manno et al., *Nature*, 2018) is a computational method for inferring the **future transcriptional state** of individual cells from a snapshot scRNA-seq experiment. It is based on the fundamental principle of mRNA biogenesis:

```
Gene → pre-mRNA (unspliced/intronic) → mRNA (spliced/exonic) → degraded
           ↑ transcription rate (β)        ↑ splicing rate (α)    ↑ degradation (γ)
```

By measuring the **ratio of unspliced (pre-mRNA) to spliced (mature mRNA)** abundance per gene per cell, the velocity field can be computed, revealing:
- Cell state transitions and differentiation trajectories
- Fate decisions in progenitor populations
- Transcriptional dynamics invisible to static snapshots

**scVelo** (Bergen et al., *Nature Biotechnology*, 2020) generalizes this concept through dynamical modeling, solving the full transcriptional kinetic equations rather than assuming steady-state.

### Why Alevin for Velocity?

Traditional velocity pipelines used **velocyto** (which requires genome-aligned BAM files, slow) or **kallisto|bustools** (pseudoalignment-based). Alevin provides a compelling alternative because:

- **Alignment-free speed**: Uses selective alignment, far faster than STAR/velocyto
- **Multi-mapper handling**: Includes multi-mapping reads, increasing sensitivity for genes with low unique k-mer fractions (Soneson et al., 2021)
- **Joint quantification**: Simultaneously quantifies spliced and unspliced features using an expanded **splici (spliced + intronic) transcriptome reference** prepared by `eisaR`
- **UMI deduplication**: Handles UMI-based deduplication natively for droplet protocols (10x, Drop-seq, inDrop)
- **Cell barcode correction**: Built-in barcode whitelisting and error correction

### The Format Problem Convert.Alevin Solves

Alevin stores its count matrices in a **proprietary binary format** (`quants_mat.gz`) that is not directly readable by Python's AnnData/scVelo ecosystem. Furthermore, the velocity-mode output encodes spliced and unspliced counts for the **same gene in a single combined matrix** using gene name suffixes (`-S` for spliced, `-U` for unspliced), which must be separated into distinct AnnData **layers** for scVelo.

## 3. Input/Output Formats & Specifications

### 3.1 Input Formats

#### Primary Input: Alevin Output Archive (`.tar.gz`)
The module accepts the complete Alevin output directory compressed as a `.tar.gz` archive. The archive must contain the following directory structure:

```
<run_name>/
├── alevin/
│   ├── quants_mat.gz          ← REQUIRED: Binary count matrix (cells × genes)
│   ├── quants_mat_cols.txt    ← REQUIRED: Gene identifiers (column names)
│   ├── quants_mat_rows.txt    ← REQUIRED: Cell barcodes (row names)  
│   ├── whitelist.txt          ← RECOMMENDED: High-confidence cell barcodes
│   ├── featureDump.txt        ← Optional: Feature-level statistics
│   ├── filtered_cb_frequency.txt  ← Optional: Filtered barcode frequencies
│   ├── MappedUmi.txt          ← Optional: Mapped UMI counts
│   └── raw_cb_frequency.txt   ← Optional: Raw barcode frequencies
├── aux_info/
│   └── meta_info.json         ← Run metadata (mapping rate, num cells)
└── cmd_info.json              ← Command-line parameters used
```

#### Binary Format Details (`quants_mat.gz`)
The `quants_mat.gz` file is **NOT** a standard gzip-compressed text file — it is a **custom binary format** developed by the Alevin/Salmon team. Key characteristics:
- Cells × Genes matrix stored in row-major binary encoding
- Contains **both** spliced (`-S`) and unspliced (`-U`) gene columns in velocity mode
- Must be parsed using the `vpolo` library (`vpolo.alevin.parser.read_quants_bin()`)
- Can alternatively be output as `.mtx` (Matrix Market Exchange) format using `--dumpMtx` flag during quantification, but the binary format is default

#### Gene Naming Convention (Velocity Mode)
When Alevin is run on a **splici (spliced + intronic) expanded reference** prepared by `eisaR`:
```
Gene column layout in quants_mat.gz (velocity mode):
  ENSG00000001234-S  →  Spliced count for gene ENSG00000001234
  ENSG00000001234-U  →  Unspliced count for gene ENSG00000001234
  ... (all genes appear twice, once with -S and once with -U suffix)
```

#### Multi-Sample Input: Text File (`.txt`)
For batch processing, the input can be a plain text file containing **newline-delimited paths** to multiple `.tar.gz` Alevin output archives:
```
/path/to/sample1_alevin_output.tar.gz
/path/to/sample2_alevin_output.tar.gz
/path/to/sample3_alevin_output.tar.gz
```

### 3.2 Output Formats

#### Primary Output: AnnData H5AD (`.h5ad`) ⭐ RECOMMENDED
The **preferred output format** for scVelo compatibility. HDF5-based file storing the complete AnnData object:

```python
# Structure of the output AnnData object:
AnnData object with n_obs × n_vars = [n_cells] × [n_genes]

adata.X                    # Spliced count matrix (cells × genes), scipy sparse CSR
adata.layers['spliced']    # Spliced (exonic) count matrix — REQUIRED by scVelo
adata.layers['unspliced']  # Unspliced (intronic) count matrix — REQUIRED by scVelo
adata.obs                  # Cell barcode DataFrame (index = barcode strings)
adata.var                  # Gene name DataFrame (index = gene IDs without -S/-U suffix)
adata.obsm['X_pca']        # PCA embeddings (if provided, optional)
adata.obsm['X_tsne']       # tSNE embeddings (if provided, optional)
```

| AnnData Component | Content | scVelo Usage |
|------------------|---------|-------------|
| `adata.X` | Spliced count matrix | Base expression matrix |
| `adata.layers['spliced']` | Exonic/mature mRNA counts | **Required** for velocity estimation |
| `adata.layers['unspliced']` | Intronic/pre-mRNA counts | **Required** for velocity estimation |
| `adata.obs` | Cell barcode index | Cell identity |
| `adata.var` | Gene name index | Gene identity |
| `adata.obsm['X_pca']` | PCA coordinates | Optional dimension reduction |
| `adata.obsm['X_tsne']` | tSNE coordinates | Optional visualization embedding |

#### Alternative Output: Loom (`.loom`)
HDF5-based columnar format compatible with both scVelo and velocyto:
```
.loom file structure:
  /matrix          # Main count matrix
  /layers/spliced  # Spliced layer
  /layers/unspliced # Unspliced layer
  /row_attrs/CellID  # Cell barcodes
  /col_attrs/Gene    # Gene names
```

| Feature | H5AD | Loom |
|---------|------|------|
| scVelo compatibility | ✅ Native | ✅ Via `scv.read()` |
| Metadata richness | ✅ Full AnnData | ⚠️ Limited |
| File size | Similar | Similar |
| Python ecosystem | ✅ Standard | ⚠️ Additional dependency |
| R compatibility | Via zellkonverter | ✅ Direct loomR |
| **Recommended for scVelo** | **✅ YES** | ⚠️ Acceptable |

## 4. Technical Architecture & Processing Logic

### 4.1 Core Processing Pipeline

The conversion follows this logical sequence:

```
Step 1: INPUT PARSING
  ├── Extract .tar.gz archive
  ├── Locate alevin/ subdirectory
  ├── Read quants_mat_cols.txt → gene list (including -S and -U suffixed genes)
  ├── Read quants_mat_rows.txt → cell barcode list
  └── Parse quants_mat.gz binary → vpolo DataFrame (cells × combined_genes)

Step 2: MATRIX SEPARATION
  ├── Identify -S suffixed columns → spliced gene set
  ├── Identify -U suffixed columns → unspliced gene set
  ├── Extract spliced submatrix
  ├── Extract unspliced submatrix
  └── Strip suffixes to create unified gene index

Step 3: ANNDATA CONSTRUCTION
  ├── Create AnnData: X = spliced matrix (scipy sparse CSR)
  ├── Assign layers: {'spliced': spliced_matrix, 'unspliced': unspliced_matrix}
  ├── Assign obs: cell barcodes DataFrame
  ├── Assign var: gene names DataFrame
  └── Optionally: assign obsm with PCA/tSNE coordinates

Step 4: OUTPUT WRITING
  ├── If h5ad: adata.write_h5ad(output_basename + '.h5ad')
  └── If loom: adata.write_loom(output_basename + '.loom')
```

### 4.2 Key Library: vpolo

The `vpolo` library (by Avi Srivastava, the Alevin developer) is the **critical dependency** for reading Alevin's binary format:

```python
from vpolo.alevin import parser

# Core function used by Convert.Alevin:
alevin_df = parser.read_quants_bin(
    base_location,    # Path to alevin output directory
    clipped=False,    # Whether to clip negative values
    mtype="data",     # Matrix type
    rmode="rust"      # Read mode (rust-based parser is faster)
)
# Returns: pandas DataFrame (cells × genes, float values)
```

**Important note**: The `read_quants_bin` function returns a **dense** pandas DataFrame, which can be memory-intensive for large datasets (see §9 Limitations).

## 9. Known Issues & Limitations

### 9.1 Memory Allocation Issues ⚠️ CRITICAL
- **Issue**: `numpy.core` memory allocation errors on large datasets
- **Root Cause**: `vpolo.read_quants_bin()` returns a **dense pandas DataFrame**, and converting combined S+U matrices to dense numpy arrays before splitting exceeds available RAM
- **Specific Error Context**: Documented in `k3yavi/vpolo` Issue #3 — observed in a Debian Docker container on AWS with 64 GB RAM allocated
- **Mitigation**: Consider removing `to_dense()` calls; use sparse matrix operations throughout; process large datasets on high-memory instances (≥128 GB)
- **Recommendation for GenePattern**: Set `job.memory=64Gb` minimum; expose memory as configurable; document RAM requirements clearly

### 9.2 Dependency Maintenance Risk
- **Issue**: `vpolo` library appears unmaintained as alevin-fry has superseded classic Alevin
- **Impact**: Future Python version compatibility may break
- **Mitigation**: Pin to tested vpolo version in Docker image; consider migrating to pyroe for alevin-fry support in future versions

### 9.3 Gene Suffix Assumption
- **Issue**: Assumes `-S`/`-U` suffixes from `eisaR`-prepared references
- **Impact**: Custom splici references with different suffixes will cause the split to fail silently (all genes in one layer)
- **Mitigation**: Expose `spliced.suffix` and `unspliced.suffix` as user-configurable parameters with clear documentation

### 9.4 Single vs Multi-Sample Handling
- **Issue**: Initial design for single alevin output; multi-sample support requires user-prepared text file
- **Impact**: No automated sample merging or batch correction
- **Mitigation**: Document clearly; subsequent modules (scVelo, ScanpyUtilities) can handle merging

### 9.5 No Ambient RNA Removal
- **Issue**: No integration with CellBender or SoupX for ambient RNA correction
- **Impact**: Downstream velocity estimates may be confounded by ambient RNA
- **Recommendation**: Advise users to run ambient RNA correction before or after conversion; this is a pre-processing consideration outside module scope

### 9.6 Version Compatibility
- **Issue**: The classic Alevin binary format used by `vpolo` differs from alevin-fry output
- **Impact**: Users who quantify with alevin-fry (recommended for modern analyses) cannot use Convert.Alevin
- **Recommendation**: Document clearly in module description; point alevin-fry users to pyroe

### 10.3 Validation & Error Handling Recommendations

```python
# Input validation checks for GenePattern module:

1. Verify .tar.gz archive contains expected files:
   - alevin/quants_mat.gz (REQUIRED)
   - alevin/quants_mat_cols.txt (REQUIRED)  
   - alevin/quants_mat_rows.txt (REQUIRED)
   - alevin/whitelist.txt (WARN if missing and filter_barcodes=true)

2. Verify gene naming:
   - Check that specified suffixes (-S, -U) are present in quants_mat_cols.txt
   - WARN if <10% of genes have expected suffixes (may not be velocity mode output)
   - ERROR if no genes have expected suffixes

3. Memory check:
   - Estimate matrix size from row/col count files before loading
   - WARN if estimated memory > 80% of job.memory allocation

4. Output validation:
   - Verify adata.layers['spliced'] and adata.layers['unspliced'] are populated
   - Log cell count, gene count, sparsity statistics
   - Verify h5ad file can be re-opened after writing
```
