## 1. 🎯 Tool Purpose & Scientific Applications

### Foundational Algorithm

The GenePattern **RandomForest** module implements the canonical Random Forest (RF) classification algorithm, originally formalized by Leo Breiman (2001, *Machine Learning*, 45:5–32). Random Forests are **ensembles of decision trees** where each tree is trained on a bootstrap sample of the data and uses a random subset of features at each split node. Classification is determined by **majority vote** across all trees.

The key mathematical properties that make RF invaluable for genomics:

| Property | Biological Significance |
|---|---|
| **Handles p >> n** | Works with thousands of genes and far fewer samples — typical in genomics |
| **Implicit feature selection** | Naturally ranks gene importance without pre-filtering |
| **No distributional assumptions** | Handles non-normal expression distributions |
| **Robust to noise** | Tolerates irrelevant features common in high-dimensional omics data |
| **No overfitting with more trees** | Increasing `n_estimators` never hurts generalization |
| **OOB error estimate** | Built-in unbiased error estimation without a held-out set |

## 2. 📁 Input/Output Formats & Requirements

### Input File Formats

#### ① GCT Format (Gene Cluster Text) — Feature Data
```
#1.2                              ← Version line (required)
7129    38                        ← [num_genes] TAB [num_samples]
Name    Description   S1  S2  S3  ...  S38    ← Header row
GENE_1  Description1  4.2 3.1 5.0 ...  2.7    ← Data rows
GENE_2  Description2  1.1 0.9 2.3 ...  3.4
...
```
- **Tab-delimited** plain text
- **Rows** = genes/features
- **Columns** = samples
- Columns 1–2: gene ID and description; Columns 3+: expression values

#### ② CLS Format (Class Label) — Target Labels
```
38 2 1                    ← [num_samples] [num_classes] [1]
# AML ALL                 ← [#] [class_0_name] [class_1_name]
0 0 0 1 1 1 0 0 ...      ← Integer class assignments per sample
```
- **Tab/space-delimited** plain text
- Line 1: sample count, class count, format indicator
- Line 2: `#` prefix followed by class names
- Line 3: integer class labels (0-indexed) for each sample

> ⚠️ **Critical Requirement:** The number of samples in the `.gct` file must **exactly match** the number of labels in the `.cls` file.

### Output File Formats

#### ③ `.pred.odf` — Prediction Results File
```
ODF 1.0
HeaderLines=7
COLUMN_NAMES: Sample   TrueClass   PredictedClass   Confidence
...
[data rows: per-sample prediction results]
```
- Output Description Format (ODF) — native GenePattern format
- Contains **true vs. predicted class** for every sample
- Includes an **accuracy score** summary
- Viewable in GenePattern's **PredictionResultsViewer** module
- Generated in **both** cross-validation and test-train modes

#### ④ `.feat.odf` — Feature Importance File
```
ODF 1.0
COLUMN_NAMES: FeatureName   ImportanceScore
GENE_ABC      0.0847
GENE_XYZ      0.0632
...
```
- Lists **feature (gene) importance scores** from the trained RF model
- Only generated in **test-train prediction mode**
- Scores reflect mean decrease in Gini impurity (or permutation importance)
- Enables downstream biomarker prioritization

### Classification Modes Summary

```
┌─────────────────────────────────────────────────────────────────┐
│                    CLASSIFICATION MODES                          │
├─────────────────────────────────────────────────────────────────┤
│  MODE 1: Cross-Validation (cross_val)                           │
│  ─────────────────────────────────────                          │
│  Inputs:  [train.gct] + [train.cls]                             │
│  Method:  Leave-One-Out Cross Validation (LOOCV)                │
│  Rationale: Prevents data leakage on single-dataset tasks       │
│  Outputs: [prefix].pred.odf                                     │
│                                                                 │
│  MODE 2: Test-Train Prediction (test_train)                     │
│  ──────────────────────────────────────────                     │
│  Inputs:  [train.gct] + [train.cls]                             │
│           [test.gct]  + [test.cls]                              │
│  Method:  Train on training set, predict on held-out test set   │
│  Outputs: [prefix].pred.odf + [prefix].feat.odf                 │
└─────────────────────────────────────────────────────────────────┘
```

> 🔬 **Scientific Rationale for LOOCV:** When only a single dataset is provided (cross-validation mode), using the same data for both training and testing would cause **data leakage** and artificially inflated accuracy. LOOCV trains on N−1 samples and tests on the held-out sample, repeating N times for N total predictions — an unbiased evaluation strategy.

## 3. ⚙️ Parameter Analysis & Usage Patterns

### Complete Parameter Catalog

#### 🗂️ GROUP A: Data Input Parameters

| Parameter | Type | Required | Description |
|---|---|---|---|
| `data_file` | `.gct` file | ✅ Always | Feature data matrix (genes × samples) |
| `cls_file` | `.cls` file | ✅ Always | Class labels for samples in `data_file` |
| `test_data_file` | `.gct` file | ⬜ test_train mode only | Test set feature matrix |
| `test_cls_file` | `.cls` file | ⬜ test_train mode only | Test set class labels |
| `pred_type` | Enum | ✅ Always | `cross_val` or `test_train` |
| `output_file_prefix` | String | ✅ Always | Prefix for output filenames |

#### 🌲 GROUP B: Forest Architecture Parameters

| Parameter | Type | Default | Range | Biological Guidance |
|---|---|---|---|---|
| `n_estimators` | int | 100 | ≥10, typically 100–1000 | More trees = more stable estimates; 500+ recommended for production |
| `max_depth` | int or None | None | None or ≥1 | `None` = unlimited; constrain to 5–20 to prevent overfitting on small datasets |
| `max_features` | str/int/float | `'sqrt'` | `'sqrt'`, `'log2'`, None, int, float | `'sqrt'` = √p features; `'log2'` = log₂(p); standard for classification |
| `criterion` | str | `'gini'` | `'gini'`, `'entropy'`, `'log_loss'` | Gini for speed; Entropy for information-theoretic interpretation |
| `max_leaf_nodes` | int or None | None | None or ≥2 | Limits tree complexity; useful for noisy genomic data |

#### 🍃 GROUP C: Node & Leaf Splitting Parameters

| Parameter | Type | Default | Range | Notes |
|---|---|---|---|---|
| `min_samples_split` | int or float | 2 | ≥2 | Minimum samples to consider splitting a node |
| `min_samples_leaf` | int or float | 1 | ≥1 | Minimum samples at any leaf; increase to smooth predictions |
| `min_impurity_decrease` | float | 0.0 | ≥0.0 | Node splits only if impurity decrease ≥ this threshold |

#### 🔀 GROUP D: Sampling & Reproducibility Parameters

| Parameter | Type | Default | Options | Notes |
|---|---|---|---|---|
| `bootstrap` | bool | True | True/False | Enables bagging; `False` = use full dataset for each tree |
| `oob_score` | bool | False | True/False | Out-of-bag accuracy estimate; requires `bootstrap=True` |
| `random_state` | int or None | None | Any int | **Critical for reproducibility**; set for deterministic outputs |
| `class_weight` | str/dict | None | `'balanced'`, `'balanced_subsample'`, dict | Use `'balanced'` for imbalanced class distributions |
| `n_jobs` | int | 1 | -1 to N | `-1` = use all available CPU cores (parallelized tree building) |

### Parameter Decision Tree for Bioinformatics Users

```
START HERE
    │
    ├─ Single dataset? ──→ pred_type = cross_val
    │                        │
    │                        └─ LOOCV handles validation automatically
    │
    └─ Two datasets? ───→ pred_type = test_train
                           │
                           └─ Requires separate train/test GCT+CLS pairs

THEN CONFIGURE:
    │
    ├─ Small dataset (n < 100 samples)?
    │     └─ ↑ n_estimators (≥500), keep max_depth=None, bootstrap=True
    │
    ├─ Large dataset (n > 1000 samples)?
    │     └─ n_estimators=100–500, consider max_depth=10–20
    │
    ├─ Imbalanced classes?
    │     └─ class_weight='balanced', oob_score=True for monitoring
    │
    ├─ Need reproducibility?
    │     └─ Set random_state to any fixed integer (e.g., 42)
    │
    └─ Multi-core system?
          └─ n_jobs=-1 to leverage all available CPUs
```

### Typical Parameter Configurations

**Configuration 1 — Leukemia (AML/ALL) Classification, Cross-Validation:**
```bash
python randomforest.py \
  --data_file all_aml_train.gct \
  --cls_file all_aml_train.cls \
  --pred_type cross_val \
  --n_estimators 500 \
  --criterion gini \
  --random_state 42 \
  --n_jobs -1 \
  --output all_aml_results
```

**Configuration 2 — Breast Cancer Test-Train Prediction:**
```bash
python randomforest.py \
  --data_file all_aml_test.gct \
  --cls_file all_aml_test.cls \
  --test_data_file all_aml_train.gct \
  --test_cls_file all_aml_train.cls \
  --pred_type test_train \
  --n_estimators 1000 \
  --max_depth 15 \
  --class_weight balanced \
  --random_state 42 \
  --output brca_results
```

## 8. ⚠️ Known Limitations & Caveats

1. **Non-determinism by default:** RF uses randomness; outputs vary between runs unless `random_state` is explicitly set. This is critical for reproducible research.

2. **LOOCV computational cost:** Leave-One-Out Cross Validation builds N separate models (where N = number of samples). For large datasets (N > 500), this can be slow. Consider k-fold CV as an alternative.

3. **Feature importance caveat:** Gini-based importance scores can be **biased toward high-cardinality features** (variables with many unique values). For unbiased importance, permutation importance is preferred (not default in this module).

4. **Memory for large GCT files:** Very large expression matrices (tens of thousands of genes × thousands of samples) require substantial RAM. Pre-filtering with `PreprocessDataset` or `ComparativeMarkerSelection` is recommended.

5. **No native cross-validation feature importance:** Feature importance (`.feat.odf`) is only generated in test-train mode, not in cross-validation mode, limiting its utility in single-dataset workflows.

6. **Class imbalance:** Default RF is sensitive to imbalanced classes. Users must explicitly set `class_weight='balanced'` for unequal class distributions.

7. **Module filename bug fixes version:** The current version is specifically tagged for filename-related bug corrections, suggesting prior issues with output file naming conventions.
