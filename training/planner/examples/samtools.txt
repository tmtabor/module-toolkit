### 1.2 Primary Purpose

SAMtools is the **de facto reference implementation and standard toolkit** for post-processing, manipulating, and analyzing high-throughput sequencing (HTS) alignment data stored in **SAM, BAM, and CRAM** formats. Originally developed as part of the **1000 Genomes Project** infrastructure, it has become an indispensable component of virtually every NGS analysis pipeline in existence.

> *"SAMtools implements various utilities for post-processing alignments in the SAM format, such as indexing, variant caller and alignment viewer."* — Original publication abstract, 2009

### 1.3 Scientific Applications

SAMtools serves as the central preprocessing and format-conversion hub in essentially all genomics domains:

| Application Domain | SAMtools Role |
|---|---|
| Whole Genome Sequencing (WGS) | Sort, index, deduplicate, QC, pileup |
| Whole Exome Sequencing (WES) | Targeted region extraction, depth, coverage |
| RNA-seq | BAM post-processing, QC statistics, format conversion |
| ChIP-seq / ATAC-seq / CUT&RUN | BAM filtering, deduplication, depth analysis |
| Single-cell Genomics | BAM manipulation, cell barcode tag operations |
| Amplicon Sequencing | Primer clipping, amplicon statistics |
| Long-read Sequencing (PacBio/ONT) | CRAM support, region extraction, statistics |
| Population Genomics | Multi-sample pileup, variant preprocessing |
| Metagenomics | Read filtering, unmapped read extraction |
| CNV Analysis | Coverage depth, pileup generation (see bcftools docs) |
| Forensic Genomics | Duplicate detection, alignment QC |

## 2. Input/Output Formats & Requirements

### 2.1 Primary Input Formats

#### SAM (Sequence Alignment/Map)
- **Type:** Human-readable tab-delimited text
- **Structure:** Header section (@HD, @SQ, @RG, @PG lines) + alignment records with **11 mandatory fields**: QNAME, FLAG, RNAME, POS, MAPQ, CIGAR, RNEXT, PNEXT, TLEN, SEQ, QUAL
- **Size:** Typically large (uncompressed); 10–100 GB for human WGS
- **Use:** Primary output format from aligners; input for initial conversion

#### BAM (Binary Alignment/Map)
- **Type:** BGZF-compressed binary version of SAM
- **Compression:** DEFLATE-based BGZF (blocked gzip); random-access capable with index
- **Size:** ~3–5x smaller than equivalent SAM; typical human WGS ~30–80 GB
- **Index:** Requires `.bam.bai` (BAI) or `.bam.csi` (CSI) for random access
- **Limitation:** BAI format cannot index chromosomes > 2²⁹ bases (~512 Mb); use CSI for scaffolded assemblies

#### CRAM (Compressed Reference-oriented Alignment Map)
- **Type:** Reference-based compression; far more compact than BAM
- **Versions:** v2.1 (lossless), v3.0 (lossless + optional lossy), v3.1 (default in htslib ≥1.21; improved compression)
- **Size:** ~40–50% smaller than BAM; requires reference genome for decoding
- **Index:** Requires `.cram.crai` index
- **Important:** Default output CRAM version changed to 3.1 in htslib 1.21+; some older tools may not support 3.1

#### FASTA/FASTQ
- **Input to:** `faidx`, `fqidx`, `calmd`, `mpileup` (reference), `import`
- **Output from:** `fasta`, `fastq` conversion commands
- **bgzip support:** bgzip-compressed + tabix-indexed FASTA fully supported via htslib

#### BED (Browser Extensible Data)
- **Used by:** `view -L`, `bedcov`, `ampliconclip`, `depth -b`, `stats -t`
- **Format:** Tab-delimited: chrom, chromStart, chromEnd (0-based, half-open intervals)

#### VCF/BCF
- **Output from:** `mpileup -v` (VCF) / `mpileup -g` (BCF, deprecated)
- **Intended use:** Pipe to `bcftools call` for variant calling

### 2.3 SAM FLAG Field Values (Critical for Filtering)

| Bit Value | Hex | Meaning |
|---|---|---|
| 1 | 0x1 | Template having multiple segments (paired-end) |
| 2 | 0x2 | Each segment properly aligned |
| 4 | 0x4 | Segment unmapped |
| 8 | 0x8 | Next segment unmapped (mate unmapped) |
| 16 | 0x10 | SEQ reverse complemented |
| 32 | 0x20 | SEQ of next segment reversed |
| 64 | 0x40 | First segment in template (read 1) |
| 128 | 0x80 | Last segment in template (read 2) |
| 256 | 0x100 | Secondary alignment |
| 512 | 0x200 | Not passing filters (QC fail) |
| 1024 | 0x400 | PCR or optical duplicate |
| 2048 | 0x800 | Supplementary alignment |

**Common Filter Combinations:**
- `-F 4` → Keep only mapped reads
- `-f 2` → Keep only properly paired reads
- `-F 1796` → Exclude unmapped + secondary + QC-fail + duplicates (standard for variant calling)

|---|---|---|---|

| CSI | `.bam.csi` | Unlimited | Large scaffolds, plant genomes, long reads |
| CRAI | `.cram.crai` | Unlimited | CRAM files |
| FAI | `.fasta.fai` | — | Reference FASTA indexing |

## 4. Detailed Parameter Documentation

### 4.1 Universal Global Options (All Commands)

| Parameter | Type | Default | Description |
|---|---|---|---|
| `--threads / -@` | `INT` | `0` | Additional worker threads (**total = value + 1**); effective for view/sort/markdup/merge/index |
| `--reference` | `filepath` | none | Reference FASTA (required for CRAM decode/encode) |
| `--input-fmt` | `string` | auto-detect | Force input format: `sam`, `bam`, `cram` |
| `--output-fmt / -O` | `string` | varies | Force output format: `SAM`, `BAM`, `CRAM` |
| `--write-index` | `flag` | off | Auto-write index alongside output (sort/view/merge/markdup) |
| `--verbosity` | `INT` | `1` | Log verbosity level (0–9) |
| `--input-fmt-option` | `OPT[=VAL]` | — | Format-specific input options |
| `--output-fmt-option` | `OPT[=VAL]` | — | Format-specific output options |

> ⚠️ **Threading Note:** `-@ N` adds N **additional** threads. Total threads = N+1. I/O-bound operations have diminishing returns above 8–16 threads. For sort, `4–8` threads with `-m 4G` is typically optimal.

**Usage Examples:**

## 6. Common Workflows & Use Cases

### 6.1 Workflow 1: Standard DNA-seq Preprocessing (GATK Best Practices)

```bash
# Step 1: Align + sort in one pipeline
bwa mem -R "@RG\tID:sample1\tSM:sample1\tPL:ILLUMINA\tLB:lib1" \
    ref.fasta R1.fastq.gz R2.fastq.gz | \
    samtools sort -@ 8 -m 4G -o sorted.bam

# Step 2: Name-sort for fixmate
samtools sort -n -@ 8 -o namesorted.bam sorted.bam

# Step 3: Fix mate information (adds ms/MC tags)
samtools fixmate -m -@ 8 namesorted.bam fixmate.bam

# Step 4: Coordinate sort again
samtools sort -@ 8 -m 4G -o coord_sorted.bam fixmate.bam

# Step 5: Mark/remove duplicates (optical dist=2500 for NovaSeq)
samtools markdup -r -d 2500 -S -s --json markdup_stats.json \
    --write-index -@ 8 coord_sorted.bam dedup.bam

# Step 6: QC reports
samtools flagstat -O json dedup.bam > flagstat.json
samtools stats -r ref.fasta dedup.bam > stats.txt
samtools idxstats dedup.bam > idxstats.txt

# Downstream: GATK HaplotypeCaller, base recalibration, etc.
```

### 6.2 Workflow 2: RNA-seq Post-processing

```bash
# STAR output is already sorted by coordinate
samtools index -@ 8 Aligned.sortedByCoord.out.bam

# QC statistics
samtools flagstat Aligned.sortedByCoord.out.bam > flagstat.txt
samtools idxstats Aligned.sortedByCoord.out.bam > idxstats.txt
samtools stats Aligned.sortedByCoord.out.bam > stats.txt
samtools coverage Aligned.sortedByCoord.out.bam > coverage_summary.txt

# Extract reads mapping to specific chromosomes
samtools view -b Aligned.sortedByCoord.out.bam chr1 chr2 chr3 -o subset.bam

# Downstream: featureCounts, StringTie, DESeq2
```

### 6.3 Workflow 3: Format Conversion & Storage Optimization

```bash
# SAM → BAM (during alignment pipeline)
samtools view -bS -@ 4 -o output.bam input.sam

# BAM → CRAM (~40-50% storage savings; requires reference)
samtools view -C -T reference.fasta \
    --output-fmt-option version=3.0 \
    --write-index -@ 4 -o output.cram input.bam

# CRAM → BAM (for tools not supporting CRAM)
samtools view -b -T reference.fasta -o output.bam input.cram

# BAM → FASTQ (for re-alignment)
samtools collate -@ 4 input.bam | \
    samtools fastq -@ 4 -1 R1.fastq.gz -2 R2.fastq.gz \
                   -0 other.fastq.gz -s singleton.fastq.gz
```

### 6.4 Workflow 4: Read Filtering & Extraction

```bash
# Keep only high-quality, properly paired, non-duplicate reads
# FLAG 1804 = 4 (unmapped) + 8 (mate unmapped) + 256 (secondary) + 512 (QC fail) + 1024 (dup)
samtools view -b -f 2 -F 1804 -q 20 -o filtered.bam sorted.bam

# Extract reads from target regions
samtools view -b -L capture_targets.bed sorted.bam -o on_target.bam

# Extract unmapped reads (for pathogen detection)
samtools view -b -f 4 -F 264 sorted.bam -o unmapped.bam

# Random subsample 10% of reads (seed=42)
samtools view -b -s 42.1 sorted.bam -o 10pct_subsample.bam
```

### 6.5 Workflow 5: Multi-sample Pileup & Variant Calling Prep

```bash
# Create sample list file
ls *.dedup.bam > bam_list.txt

# Generate multi-sample BCF for variant calling
samtools mpileup -f ref.fasta \
    -q 20 -Q 20 -C 50 \
    -d 10000 \
    --output-tags DP,AD \
    -u -b bam_list.txt | \
    bcftools call -mv --threads 8 -Oz -o variants.vcf.gz

bcftools index variants.vcf.gz
```

### 6.6 Workflow 6: Coverage Analysis

```bash
# Per-base depth (all positions)
samtools depth -a -Q 20 -q 20 -o depth_per_base.txt sorted.bam

# Coverage summary per chromosome
samtools coverage sorted.bam > chromosome_coverage.txt

# Read depth per target BED region
samtools bedcov capture_targets.bed sorted.bam > region_depth.txt

# Comprehensive QC (feeds MultiQC)
samtools stats -r ref.fasta -d -t capture_targets.bed sorted.bam > bamstats.txt
```

### 6.7 Workflow 7: Amplicon Sequencing

```bash
# Clip primer sequences from reads using primer BED
samtools ampliconclip -b primers.bed --hard-clip \
    -o clipped.bam -r clip_stats.txt input.bam

# Generate amplicon-specific statistics
samtools ampliconstats -F QCFAIL,DUP primers.bed clipped.bam > amplicon_stats.txt

# Plot amplicon coverage (if matplotlib available)
plot-ampliconstats amplicon_stats.txt
```

### 7.2 samtools Unique Strengths

1. **Reference implementation**: Co-authored the SAM/BAM/CRAM specification; ultimate authority
2. **Ubiquity**: Installed on essentially every HPC cluster and bioinformatics server worldwide
3. **Streaming pipeline support**: Designed for Unix pipe chaining; minimal memory overhead
4. **CRAM support**: Native, full-featured CRAM v2.1/3.0/3.1 read/write
5. **mpileup**: The standard multi-sample pileup engine; used by bcftools for population variant calling
6. **tview**: Only major tool with a terminal-based alignment viewer (no GUI required)
7. **Active development**: Frequent releases, responsive maintainers (Wellcome Sanger)
8. **Complete format coverage**: SAM+BAM+CRAM+FASTA+FASTQ+BED+VCF all handled natively

### 7.3 Known Limitations

- **BAI index**: Cannot index chromosomes >512 Mb (use CSI: `samtools index -c`)
- **Duplicate metrics**: Less detailed than Picard's `CollectDuplicateMetrics` (but `--json` flag helps)
- **Thread scaling**: Sort threading less efficient than sambamba at very high core counts (>16)
- **Variant calling**: mpileup output should be piped to bcftools call; not a standalone caller
- **Visualization**: tview is text-only; IGV or deeptools needed for publication figures
- **No BAM validation**: Unlike Picard, does not perform exhaustive format validation by default

## 9. GenePattern Integration Recommendations

### 9.1 Module Architecture Strategy

Given samtools' wide command set, we strongly recommend a **multi-module approach** rather than one monolithic module:

| Proposed Module Name | Subcommand(s) | Priority | Use Case |
|---|---|---|---|
| `samtools.View` | `view` | 🔴 Critical | SAM/BAM/CRAM conversion + filtering |
| `samtools.Sort` | `sort` | 🔴 Critical | Coordinate/name sorting |
| `samtools.Index` | `index` | 🔴 Critical | BAM/CRAM indexing |
| `samtools.Flagstat` | `flagstat` | 🔴 Critical | Quick QC statistics |
| `samtools.Stats` | `stats` | 🟠 High | Comprehensive QC report |
| `samtools.Markdup` | `fixmate` + `markdup` | 🟠 High | Duplicate marking pipeline |
| `samtools.Depth` | `depth` / `coverage` | 🟠 High | Coverage analysis |
| `samtools.Mpileup` | `mpileup` | 🟡 Medium | Variant calling preparation |
| `samtools.Merge` | `merge` | 🟡 Medium | Multi-BAM merging |
| `samtools.Faidx` | `faidx` | 🟡 Medium | FASTA reference indexing |

### 9.4 Critical Integration Considerations

1. **Index File Co-location**: BAM `.bai` index must reside in the same directory as the BAM file. GenePattern file handling must ensure this. Use `--write-index` flag or pair commands appropriately.

2. **Reference Genome Handling**: CRAM operations require a reference genome. This should be a separate, versioned input parameter. Consider linking to commonly used assemblies (GRCh38, GRCh37/hg19, mm10, mm39).

3. **Threading**: GenePattern job resources should expose thread count parameter and map it to `-@` (note: `-@ N` = N *additional* threads, so total = N+1).

4. **Temporary File Space**: `samtools sort` creates temporary files of up to 3x input size. GenePattern jobs need sufficient scratch space; use `-T` to specify scratch path.

5. **Streaming Pipelines**: For complex pipelines (e.g., align → sort → markdup), consider a combined workflow module to avoid writing large intermediate files.

6. **Output Naming Convention**: Establish consistent naming: `{sample}.sorted.bam`, `{sample}.dedup.bam`, `{sample}.flagstat.json`, etc.

7. **Error Handling**: samtools exits with code 1 on errors and prints to stderr. GenePattern wrapper should capture stderr and surface it in job logs.

8. **GenePattern Existing Context**: GenePattern already references SAMtools in RNA-seq documentation and has Picard wrappers. samtools modules should complement (not duplicate) Picard functionality — use samtools for speed-sensitive operations (sort, view, index) and Picard for detailed metrics.

9. **Docker Image Versioning**: Pin exact image digest (not just tag) for reproducibility: `quay.io/biocontainers/samtools:1.21--h50ea8bc_0@sha256:...`

10. **CRAM Default Change Warning**: htslib ≥1.21 defaults to CRAM v3.1 output. Some downstream tools may not support v3.1. Use `--output-fmt-option version=3.0` for maximum compatibility, or document version behavior clearly.
