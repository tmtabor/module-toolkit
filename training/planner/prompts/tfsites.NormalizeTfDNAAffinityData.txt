### Scientific Purpose

`NormalizeTfDNAAffinityData` is a **data preprocessing module** that converts raw TF-DNA binding affinity measurements (e.g., from SELEX-seq, HT-SELEX, or Protein Binding Microarray/PBM experiments) into a standardized **relative affinity dataset**, where every k-mer sequence receives a normalized score on a **0.0 to 1.0 scale**. The normalization anchor is the sequence with the **highest raw affinity value that also conforms to the IUPAC-defined core binding site definition** of the transcription factor. This normalized output is the required standard input format for all downstream TFSites binding site prediction, scanning, and annotation modules.

## 2. 🏛️ Biological & Scientific Background

### The Farley Lab Paradigm: Enhancer Suboptimization

This module is deeply rooted in a research paradigm pioneered by the Farley Lab (UCSD). Their landmark work demonstrates that developmental enhancers operate via **"suboptimization"** — their TF binding sites have deliberately *reduced* affinity relative to the theoretical optimum. This suboptimality is not a flaw but a feature: it encodes **tissue-specificity and precise gene expression levels**. Single-nucleotide variants that increase TF affinity toward the optimum cause gain-of-function defects and disease phenotypes (e.g., polydactyly, cardiac malformations).

**This motivates the 0–1 relative affinity scale**: a normalized score of 1.0 represents the maximum achievable biological binding, while developmental enhancers typically operate with sites scoring ≤0.3–0.6.

### Key Literature

| Reference | Relevance |
|---|---|
| **Farley EK et al. (2015). *Science* 350(6258):aac6948** | Foundational "suboptimization" paper — establishes the biological need for relative affinity quantification |
| **Farley EK et al. (2016). *PNAS* 113:6508–6513** | Uses relative affinity classes (low ≤0.3; high ≥0.6) to classify ETS/GATA sites in Ciona enhancers |
| **Lim F, Solvason JJ, Farley EK et al. (2024). *Nature* 626(7997):151–159** | Demonstrates affinity-optimizing variants disrupt vertebrate development; validates the 0–1 framework |
| **Slattery M et al. (2014). SELEX-seq methods. *Methods Mol Biol*** | Background on SELEX-seq k-mer affinity estimation |

### Analogy with `tfsites.GenerateMotifDatabase`

These two modules serve **parallel roles** in the TFSites ecosystem. `GenerateMotifDatabase` normalizes data from **JASPAR Position Frequency Matrices** (the MFI value of the highest-scoring k-mer matching the IUPAC definition → 1.0). `NormalizeTfDNAAffinityData` performs the **identical normalization logic** but for raw **experimentally-derived affinity data** (SELEX, PBM, HT-SELEX). Together they constitute the two entry points into the normalized affinity database format.

## 4. 📥📤 Input/Output Format Specifications

### Input: Raw Affinity Data File

| Attribute | Specification |
|---|---|
| **Format** | Tab-separated (TSV) or comma-separated (CSV) |
| **Structure** | Two-column table: `[sequence] [raw_affinity_value]` |
| **Column 1** | DNA k-mer sequence (typically 6–12 bp, A/T/G/C only) |
| **Column 2** | Raw numeric affinity/signal value (e.g., MFI from PBM, enrichment from SELEX) |
| **Header** | Optional header row |
| **Data completeness** | Ideally exhaustive k-mer enumeration for the given k length |

**Example Input (`raw_ets_affinity.tsv`):**
```
sequence    raw_affinity
CCGGAAGT    1420.7
CCGGAAGG    1380.2
CAGGAAGT    850.4
ACGGAAGT    310.8
TTGCATGC    45.2
```

**Data Source Types Accepted:**
- 🧪 **SELEX-seq**: k-mer relative enrichment ratios across selection cycles
- 🔬 **HT-SELEX**: High-throughput SELEX enrichment counts
- 📊 **PBM (Protein Binding Microarray)**: Median Fluorescence Intensity (MFI) per 8-mer
- 🔭 **Custom**: Any quantitative TF-DNA binding measurement

### Core Binding Site Definition (Parameter — Not a File)

```
IUPAC string: e.g., RGGAAWY (ETS), WGATAR (GATA), ACAATGG (SOX)

IUPAC Code Reference:
  R = A or G    Y = C or T    W = A or T
  S = C or G    K = G or T    M = A or C
  B = C, G, T   D = A, G, T   H = A, C, T
  V = A, C, G   N = any base
```

Only sequences matching this pattern are candidates for the normalization reference maximum.

### Output: Normalized Affinity Data File

| Attribute | Specification |
|---|---|
| **Format** | Tab-separated values (TSV) |
| **Structure** | Two-column table: `[sequence] [relative_affinity]` |
| **Value range** | 0.0 (no measurable affinity) → 1.0 (maximum conforming affinity) |
| **Content** | ALL sequences from original dataset, each rescaled |
| **Downstream use** | Direct input to `tfsites.AffinityScan`, `tfsites.DefineTfSites` |

**Example Output (`normalized_ets_affinity.tsv`):**
```
sequence    relative_affinity
CCGGAAGT    1.000
CCGGAAGG    0.972
CAGGAAGT    0.599
ACGGAAGT    0.219
TTGCATGC    0.032
```

### Normalization Algorithm (Pseudocode)

```python
# 1. Load raw affinity data
raw_data = load_tsv(input_file)  # dict: {sequence: raw_value}

# 2. Filter sequences conforming to IUPAC core binding site definition
conforming = {seq: val for seq, val in raw_data.items() 
              if matches_iupac(seq, core_definition)}

# 3. Find normalization anchor
max_value = max(conforming.values())

# 4. Normalize all sequences relative to max
normalized = {seq: val / max_value for seq, val in raw_data.items()}

# 5. Write output
write_tsv(output_file, normalized)
```

## 8. 📊 Comparative Analysis

### TFSites Approach vs. Alternative Methods

| Tool/Approach | Method | Key Difference from `NormalizeTfDNAAffinityData` |
|---|---|---|
| **MEME Suite (FIMO)** | PWM-based log-odds scoring | Uses statistical position-independence model; cannot represent non-additive k-mer contributions |
| **SELEX R Package** | Kullback-Leibler divergence cycle enrichment | Generates k-mer enrichment tables; does NOT apply IUPAC-constrained normalization — **complementary upstream step** |
| **DeepBind** | Deep learning (CNN) affinity prediction | Learns from raw data but no explicit 0-1 normalization or IUPAC constraint |
| **UniPROBE DB** | Pre-normalized PBM 8-mer data | Static database; `NormalizeTfDNAAffinityData` allows user-defined TFs and custom normalization |
| **JASPAR + tfsites.GenerateMotifDatabase** | PWM-derived k-mer scoring → normalized | Parallel path: only for TFs in JASPAR; this module handles experimental affinity data |

### Unique Technical Advantages

1. **🎯 IUPAC-Constrained Normalization**: The anchor (1.0) must be a biologically valid binding site sequence — prevents non-specific sequences from distorting the scale.
2. **📈 Full k-mer Affinity Preservation**: Unlike PWMs that average across positions, preserves true binding affinity for every unique k-mer combination, capturing positional interdependencies.
3. **🔗 Ecosystem Integration**: Output is natively compatible with every downstream `tfsites.*` module without format conversion.
4. **🧩 Data Source Agnostic**: Accepts any numeric affinity measurement type (PBM MFI, SELEX enrichment, fluorescence, etc.).
5. **📐 Biologically Grounded Scale**: The 0–1 relative scale directly maps to the "suboptimization" paradigm — low-affinity sites (≤0.3) vs. high-affinity sites (≥0.6) match biologically characterized functional categories.

## 9. ⚠️ Limitations & Caveats

| Limitation | Impact | Mitigation |
|---|---|---|
| **Requires pre-existing experimental data** | Cannot be used without raw SELEX/PBM/HT-SELEX data | Pair with `tfsites.GenerateMotifDatabase` for JASPAR-covered TFs |
| **Draft release status** | May lack production robustness, full error handling, complete documentation | Validate with known datasets (ETS, GATA) before production use |
| **IUPAC definition quality dependent** | Incorrect IUPAC pattern yields wrong normalization anchor | Use established consensus sequences from literature |
| **Tight tfsites ecosystem coupling** | Output primarily useful within tfsites workflow | Consider exporting normalized data in PWM format for broader compatibility |
| **k-mer length not standardized** | Input data may use various k lengths | Document expected k-mer length for each TF |
| **Strand handling** | May not explicitly handle reverse complement aggregation | Verify convention: is reverse complement of a k-mer treated as the same or separate entry? |

### Critical Implementation Notes

1. **IUPAC Pattern Validation**: The `core_binding_site_definition` parameter must be validated client-side to ensure it contains only valid IUPAC characters and matches the k-mer length of the input data. A mismatch here produces silent incorrect results.

2. **Max-Affinity Conformer Must Exist**: If no sequence in the input data matches the IUPAC pattern, the tool must throw a **clear, descriptive error** (not divide-by-zero). This edge case requires explicit handling.

3. **Delimiter Auto-detection**: Consider auto-detecting tab vs. comma from file extension (`.tsv` vs. `.csv`) to improve user experience.

4. **Header Auto-detection**: Check whether first row contains non-numeric values to auto-detect header presence.

5. **Output Format Consistency**: Verify that the exact output column names and delimiter match the expected input format of `tfsites.AffinityScan` and `tfsites.DefineTfSites`.

6. **Value Scaling Note for Documentation**: Emphasize that values > 1.0 are impossible by design; values exactly equal to 1.0 represent the maximum-affinity conforming sequence; scores of 0.0 represent background/non-binding sequences.
