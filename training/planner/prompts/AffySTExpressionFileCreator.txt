### 1.2 Scientific Purpose

AffySTExpressionFileCreator addresses a critical gap in Affymetrix microarray preprocessing: the inability of older tools to handle **Sense Target (ST)** whole-transcript arrays. These arrays, introduced beginning around 2006, use a fundamentally different probe design than the legacy 3′-biased IVT arrays and therefore require a distinct processing pipeline.

The module creates a **gene-level expression matrix in GCT format** from raw Affymetrix ST array CEL files using the industry-standard **Robust Multi-Array Average (RMA)** algorithm — implemented via the Bioconductor `oligo` package — thereby bridging raw probe intensities to analysis-ready gene expression values suitable for the entire GenePattern analytical ecosystem.

### 1.3 Why This Module Exists: The ST Array Problem

Affymetrix introduced the **Whole Transcript (WT/ST) array design** to overcome a fundamental limitation of 3′-biased IVT arrays:

| Feature | IVT Arrays (Legacy) | ST Arrays (Modern) |
|---|---|---|
| **Target prep** | In vitro transcription amplification | Random primer-based whole-transcript amplification |
| **Coverage** | ~3′ end of transcripts only (~25% of transcript) | Entire transcript body |
| **Probe distribution** | One location per transcript | Multiple probesets distributed across transcript |
| **Probe sharing** | No — unique probesets | Yes — probes shared across probesets |
| **GenePattern tool** | `ExpressionFileCreator` | **`AffySTExpressionFileCreator`** |
| **Bioconductor tool** | `affy` package | `oligo` package |

The probe-sharing architecture of ST arrays is exactly why `ExpressionFileCreator` explicitly documents it **cannot** process ST/Exon/HTA arrays. The underlying `affy` R package was not designed for arrays where probes are shared between probesets.

### 2.3 Core R Processing Logic

```r
# Pseudocode of internal processing pipeline
library(oligo)
library(AnnotationDbi)

# 1. Read CEL files (oligo auto-detects array type from CEL headers)
raw_data <- read.celfiles(cel_file_list)

# 2. Apply RMA (conditionally based on parameters)
eset <- rma(raw_data,
            normalize = normalize_param,    # quantile normalization
            background = background_param)  # convolution background model

# 3. Extract expression matrix (log2 transformed values)
expr_matrix <- exprs(eset)  # rows=probesets, cols=samples

# 4. Annotate with gene symbols (if annotation=yes)
probe_ids <- featureNames(eset)
annot <- AnnotationDbi::select(array.db,
                               keys = probe_ids,
                               columns = c("SYMBOL", "GENENAME"))

# 5. Apply CLM-based sample ordering/renaming (if provided)
# 6. Write GCT output file
# 7. Optionally write CLS file from CLM phenotype data
```

## 3. Input/Output Format Specifications

### 3.1 Input Files

#### 3.1.1 CEL Files (Required)
- **Parameter name**: `cel.files`
- **Description**: Raw Affymetrix array intensity files
- **Accepted Formats**:

| Format | Description | Notes |
|---|---|---|
| `.CEL` | Uncompressed binary CEL file | Standard format from Affymetrix scanners |
| `.CEL.gz` | GZ-compressed CEL file | Saves ~50-70% disk space |
| `.zip` | ZIP archive of CEL files | Most common bulk submission format |
| `.tar` | TAR archive of CEL files | Common on Linux systems |
| `.tar.gz` | GZ-compressed TAR archive | Very common for GEO downloads |
| `.tar.xz` | XZ-compressed TAR archive | High compression ratio |
| `.tar.bz2` | BZ2-compressed TAR archive | Alternative compression |
| Directory | Server directory path | Requires server directory-input support |

> ⚠️ **Critical Constraint**: All CEL files in a single run **must be from the same array type**. Mixing array types in one job will cause an error. Array type is auto-detected from CEL file binary headers.

> ⚠️ **Directory inputs**: Directories are **recursively searched** for CEL files, but ZIP/TAR archives found within directory inputs are **NOT** further processed.

#### 3.1.2 CLM File (Optional)
- **Parameter name**: `clm.file`
- **File extension**: `.clm`
- **Description**: GenePattern CEL Label Map — controls sample naming, ordering, and class annotation

**CLM File Format (Tab-Delimited)**:
```
scan_name    sample_name    phenotype_label
sample1.CEL  Control_1      control
sample2.CEL  Control_2      control
sample3.CEL  Treated_1      treatment
sample4.CEL  Treated_2      treatment
```

| Column | Content | Required |
|---|---|---|
| Column 1 | CEL scan filename (with or without `.CEL` extension) | Yes |
| Column 2 | Desired sample name in output GCT | Yes |
| Column 3 | Phenotype/class label | Optional — if present, triggers CLS file generation |

> 💡 **Best Practice**: Always provide a CLM file if sample ordering matters, as without it the order is determined by filesystem processing order which can be unpredictable.

### 3.2 Output Files

#### 3.2.1 GCT File (Primary Output)
- **Format**: GenePattern Gene Cluster Text (`.gct`)
- **Structure**:
```
#1.2
[num_genes]    [num_samples]
Name    Description    Sample1    Sample2    ...    SampleN
1007_s_at    DDR1    8.234    7.891    ...    9.012
1053_at      RFC2    5.123    5.456    ...    5.789
...
```

| Element | Description |
|---|---|
| Row 1 | `#1.2` version header |
| Row 2 | Number of genes (rows) and samples (columns) |
| Row 3 | `Name` \| `Description` \| sample names |
| Rows 4+ | Probeset ID \| Gene symbol/description \| log2 expression values |

> **Note on values**: All expression values are **log2 transformed** by the RMA algorithm, typically ranging from ~3 (low/absent expression) to ~14 (high expression) on a log2 scale (i.e., 8–16,384 on linear scale).

#### 3.2.2 CLS File (Conditional Secondary Output)
- Generated only when the CLM file contains phenotype/class labels
- **Format**: Standard GenePattern CLS (Class Label) file
- Enables direct input to classification and differential expression modules

## 4. Parameter Analysis & Documentation

### 4.1 Complete Parameter Table

| Parameter | Type | Default | Required | Values | Description |
|---|---|---|---|---|---|
| `cel.files` | File input | — | **YES** | `.CEL`, `.zip`, `.tar*`, directory | One or more CEL files or archives from Affymetrix ST arrays |
| `clm.file` | File input | — | No | `.clm` | Sample descriptor file for ordering, renaming, and phenotype annotation |
| `normalize` | Boolean | `yes` | No | `yes` / `no` | Apply quantile normalization across arrays during RMA |
| `background.correct` | Boolean | `yes` | No | `yes` / `no` | Apply convolution-based background correction during RMA |
| `annotation` | Boolean | `yes` | No | `yes` / `no` | Annotate probeset IDs with gene symbols in output GCT |
| `output.file.name` | String | Auto | No | Any valid filename string | Base name for output GCT (and optionally CLS) file |

### 4.2 Parameter Group Analysis

**Group 1 — Data Inputs** *(High Priority)*
- `cel.files` — The only strictly required parameter
- `clm.file` — Highly recommended for reproducible, ordered output

**Group 2 — Preprocessing Options** *(Medium Priority)*
- `normalize` + `background.correct` — Together control the full RMA pipeline
- Setting **both to `no`** produces raw probe-level summaries (useful for QC or custom normalization workflows)
- Setting **`normalize=no`** with **`background.correct=yes`** gives background-corrected but not inter-array normalized data

**Group 3 — Output Configuration** *(Low Priority)*
- `annotation` — Recommended to keep `yes` for analysis-ready output with gene symbols
- `output.file.name` — Set to a meaningful experiment name for traceability

### 4.3 Parameter Interaction Rules

```
Key Parameter Interactions:

1. normalize=no AND background.correct=no
   → Raw, un-preprocessed intensity summaries
   → USE CASE: Custom normalization pipelines, QC diagnostics
   → WARNING: Not suitable for direct differential expression analysis

2. normalize=yes AND background.correct=yes (DEFAULT)
   → Full RMA preprocessing — recommended for all standard analyses
   → Output is log2 normalized, inter-array comparable

3. clm.file provided AND has 3 columns
   → Generates both output.gct AND output.cls
   → Enables seamless input to ComparativeMarkerSelection, GSEA

4. annotation=no
   → Output GCT uses raw Affymetrix probeset IDs as row identifiers
   → May be preferable when using external annotation pipelines
   → Required if array type is not supported by built-in annotation DBs
```

## 5. Scientific Algorithm: RMA Deep Dive

### 5.1 RMA Processing Pipeline

The RMA algorithm (Irizarry et al., *Biostatistics*, 2003) consists of three sequential steps applied by the `oligo` package:

```
Raw PM Probe Intensities (from CEL files)
         │
         ▼
┌─────────────────────────────────────────────────────┐
│  STEP 1: BACKGROUND CORRECTION                       │
│  Model: PM = signal(S) + background(B)               │
│  S ~ exponential(α), B ~ Normal(μ,σ²)               │
│  → Convolution model estimates true signal           │
│  → Removes optical noise and non-specific binding    │
└─────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────┐
│  STEP 2: QUANTILE NORMALIZATION                      │
│  Goal: Make all array distributions identical        │
│  Method: Replace each value with the mean of         │
│          equivalent quantile ranks across arrays     │
│  → Removes between-array technical variation         │
│  → Assumes most genes not differentially expressed   │
└─────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────┐
│  STEP 3: SUMMARIZATION (Median Polish)               │
│  Model: log2(PMij) = μi + αj + εij                  │
│  μi = probeset effect (= expression estimate)        │
│  αj = probe affinity effect                          │
│  εij = residual error                                │
│  → Collapses probe-level to probeset-level values    │
│  → Robust to outlier probes                          │
└─────────────────────────────────────────────────────┘
         │
         ▼
Log2-transformed, normalized expression matrix (GCT)
```

### 5.2 RMA vs. Alternative Algorithms

| Algorithm | Log2 Output | Between-Array Norm | Handles ST | Speed |
|---|---|---|---|---|
| **RMA** (used here) | Yes | Quantile | ✅ via `oligo` | Fast |
| GCRMA | Yes | Quantile | ❌ (not in `oligo`) | Slower |
| MAS5 | No (linear) | Per-probe scaling | ❌ | Moderate |
| dChip | No (linear) | Invariant set | ❌ | Slow |

> ⚠️ **Important Downstream Note**: Unlike `ExpressionFileCreator` which *removes* the log2 transformation before writing the GCT, `AffySTExpressionFileCreator` **preserves log2 values in the output**. This is critical for choosing appropriate downstream modules.

## 10. Known Issues, Limitations & Caveats

### 10.1 Critical Caveats

| Issue | Impact | Mitigation |
|---|---|---|
| **BETA status** | Potential bugs, limited QA | Test with known datasets before production use |
| **Log2 output** | May break downstream modules expecting linear scale | Check each downstream module's data scale expectations |
| **Single array type per run** | Cannot mix Human Gene 1.0 with Mouse Gene 2.0 in one job | Run separate jobs per array type |
| **Unpredictable sample ordering** | Column order in GCT may differ from file input order | Always provide CLM file for ordered experiments |
| **Directory-level ZIP exclusion** | ZIP/TARs in directory inputs are skipped | Provide archives directly as `cel.files` input |

### 10.2 Statistical Limitations

| Limitation | Detail |
|---|---|
| **Minimum array count for RMA** | RMA normalization is unreliable with < 3-4 arrays; ideally use ≥ 6. Single-array normalization is technically possible but statistically unsound. |
| **Global normalization assumption** | Quantile normalization assumes most genes are NOT differentially expressed. This may fail in extreme conditions (e.g., >30% differentially expressed genes). |
| **No technical replicate handling** | Module applies uniform RMA; batch correction must be done as a separate downstream step |
| **No exon-level summarization option** | The module produces gene/transcript-level (core probe set) summaries only; exon-level splicing analysis requires additional tools |

### 10.3 Version-Specific Notes

- **Version 0.14 (BETA)**: Only available on GenePattern Team-hosted servers; not yet released for all platforms
- **Docker image :2**: Updated runtime environment improving compatibility with newer `pd.*` Bioconductor packages
- **Older R version**: The base container uses R 3.x which may have limited compatibility with the most recently added Clariom S/D array annotation packages
